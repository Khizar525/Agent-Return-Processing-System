# ADR-001 — Agent Topology: Manager + Handoff Hybrid Pattern

**Status:** Accepted  
**Date:** 2026  
**Owner:** Project Lead  

---

## Context

We need to handle multiple distinct customer support intents (returns, tracking,
billing, general inquiries, escalations) within a single conversational flow.
Each intent requires different tools, business logic, and failure modes.

The key question: should the Triage Orchestrator always retain control and call
specialists as sub-tools (pure Manager pattern), or should it hand off full
control to specialists when needed (pure Handoff pattern)?

---

## Decision

**Adopt the Manager + Handoff Hybrid pattern with keyword-first classification.**

- **Handoff** for flows that require full specialist ownership with complex
  multi-step tool use: `return_request` → PolicyAgent, `edge_case` → EscalationAgent.
- **Deterministic tool dispatch** for narrow, single-step lookups where context must
  stay with the Orchestrator: tracking lookups, FAQ lookups, return eligibility checks.

### Tool Execution Architecture

The OpenAI Agents SDK's `Runner.run()` consumes tool call results internally and
never exposes them to the caller. When the LLM calls a tool during reasoning, those
results feed back into the LLM's context but are lost from the caller's perspective.

**Solution:** Separate classification from tool execution.

```
Message → Keyword Classifier (primary, deterministic)
       → LLM Classifier (enrichment, optional)
       → _dispatch_table(intent) → Tool Execution (deterministic code)
       → Structured Response
```

The LLM is used ONLY for intent classification (its strength). Tools are executed
deterministically in code (code's strength). This avoids the SDK limitation entirely.

---

## Rationale

| Concern | Pure Manager | Pure Handoff | Hybrid (chosen) |
|---------|-------------|--------------|-----------------|
| Context retention | ✅ Always | ❌ Lost on handoff | ✅ Retained where needed |
| Specialist autonomy | ❌ None | ✅ Full | ✅ Where appropriate |
| Debugging / tracing | ✅ Single agent | ❌ Complex chain | ✅ Clear handoff points |
| Token cost | ❌ High (full history passed) | ✅ Low | ✅ Balanced |
| Escalation context bundle | ❌ Hard to compile | ✅ Natural | ✅ Natural |
| Tool result exposure | ❌ Lost by SDK | N/A | ✅ Deterministic dispatch |

## Agent Routing Table

| Intent | Pattern | Route To | Tool Executed | Rationale |
|--------|---------|----------|---------------|-----------|
| `return_request` | Tool call | `check_return_policy` | Yes — eligibility check | Deterministic, no LLM needed |
| `order_status` | Tool call | `tracking_lookup` | Yes — tracking status | Single tool call, no specialist needed |
| `billing_dispute` | Handoff | BillingAgent | No — specialist ownership | Requires specialist access to billing API |
| `general_inquiry` | Tool call | `faq_lookup` | Yes — FAQ search | Single tool call, no specialist needed |
| `edge_case_escalate` | Handoff | EscalationAgent | No — human escalation | Full context bundle compilation required |

## Keyword Classification Rules

Primary intent detection uses deterministic keyword matching (zero cost, zero latency):

| Intent | Trigger Keywords |
|--------|-----------------|
| `edge_case_escalate` | sue, lawyer, attorney, court, legal action, litigation |
| `return_request` | return order, return item, send back, damaged, broken, wrong item, replace my |
| `billing_dispute` | charged, billing, invoice, transaction, overcharged, double charged |
| `order_status` | track, where is, delivery, shipped, tracking, order status |
| `general_inquiry` | (fallback — anything not matched above) |

## LLM Provider Compatibility

Free-tier providers (OpenRouter) have two limitations vs the official OpenAI API:

1. **Model names with `/`** (e.g. `openai/gpt-oss-120b:free`) are misinterpreted
   as provider prefixes by MultiProvider. **Fix:** Set `unknown_prefix_mode="model_id"`.
2. **`response_format`** (structured output) cannot be combined with `tools` in the
   same request. **Fix:** Strip `output_schema` when tools/handoffs are present.

Both patches are applied in `main.py` at startup.

---

## Consequences

- Every agent must return outputs matching the interface spec in
  `/docs/tool_interface_spec.md`.
- Session state must be propagated through every handoff via the `context`
  parameter — never rely on in-memory state.
- The Escalation Agent must always receive the full `agent_chain` so the
  human queue has complete context.
- Tool dispatch is deterministic — adding a new intent requires updating
  `_KEYWORD_RULES` and `_dispatch_tool()` in `triage_orchestrator.py`.

---

## Alternatives Considered

**Pure Manager pattern** — rejected because the Resolution Agent needs to
execute multiple sequential tool calls (validate → refund → label → notify)
that are cleaner with full specialist ownership.

**Pure Handoff pattern** — rejected because simple lookups (tracking, FAQ)
don't justify the overhead of a full agent handoff and context rebuild.

**LLM-dependent tool execution** — rejected because the OpenAI Agents SDK's
`Runner.run()` consumes tool call results internally and never exposes them
to the caller. The free model also produces malformed JSON intermittently,
making LLM-dependent tool calls unreliable.
